### 🧪 API 벤치마킹 핵심 테스트 시나리오

#### 🟢 Phase 1: 기본 성능 및 대용량 전송 테스트 (Baseline & Payload)

가장 단순한 형태의 데이터 요청을 통해 순수 프로토콜 간의 '데이터 직렬화/역직렬화(포장 및 해제) 속도'와 메모리 한계치를 측정합니다.

* **TC 1. 단일 레코드 단순 조회 (Baseline)**
* **대상:** 우측 상단의 노란색 `olist_products_dataset` 테이블에서 특정 `product_id` 1건만 조회.
* **측정 목적:** 조인(Join)이 전혀 없는 상태에서 REST(JSON)와 gRPC(Protobuf 바이너리) 간의 순수한 응답 속도 기준점을 잡습니다.
<br/>

* **TC 2. 대용량 페이로드 반환 (Memory Limit)**
* **대상:** 중앙의 붉은색 `olist_orders_dataset` 테이블에서 특정 고객의 주문 1,000건을 한 번에 조회.
* **측정 목적:** 데이터 크기가 MB 단위로 커질 때, 무거운 텍스트를 생성하는 REST/GraphQL과 이진법으로 압축하는 gRPC 간의 서버 메모리 소비량 차이를 비교합니다.

---

#### 🟡 Phase 2: 관계형 데이터 병목 테스트 (N+1 및 오버패칭)

ERD의 1:N 관계(화살표 방향)를 활용하여 프론트엔드가 겪는 흔한 데이터 호출 문제들을 백엔드가 어떻게 처리하는지 측정합니다.

* **TC 3. 1:N 얕은 조인 (REST 언더패칭 검증)**
* **대상:** `order_id`를 매개로 중앙의 `orders`(붉은색)와 연결된 상단의 `payments`(회색), 좌측의 `reviews`(보라색)를 동시에 조회.
* **측정 목적:** REST 아키텍처에서 여러 번 API를 호출해야만 화면을 구성할 수 있는 병목 현상과, 이를 한 번에 가져오는 GraphQL/gRPC의 효율성을 비교합니다.

<br/>

* **TC 4. 부분 필드 추출 (GraphQL 오버패칭 해결)**
* **대상:** 좌측의 `olist_order_reviews_dataset`(보라색)에서 500건을 조회하되, 전체 데이터가 아닌 `review_score`와 `review_comment_title` 딱 2개 컬럼만 요청.
* **측정 목적:** 불필요한 데이터를 모두 던져주는 REST 대비, 프론트엔드가 요구한 것만 골라내는 GraphQL의 네트워크 대역폭 절감 효과를 증명합니다.


---

#### 🔴 Phase 3: 극한의 서버 부하 테스트 (Deep Join & Transaction)

다이어그램의 끝에서 끝까지 도달하는 복잡한 연산을 통해 각 서버의 CPU가 어떻게 혹사당하는지 관찰합니다. 이 벤치마킹의 하이라이트입니다.

* **TC 5. 극한의 7+ 다중 테이블 조인 (GraphQL CPU 폭주 테스트)**
* **대상:** 다이어그램 전체 관통. `orders`(중앙) -> `order_items`(주황색) -> `products`(노란색) 및 `sellers`(초록색) -> `geolocation`(파란색)까지 이어지는 모든 경로를 한 번의 쿼리로 묶어서 대시보드용으로 조회.
* **측정 목적:** 이 깊고 복잡한 그래프 구조를 해석하기 위해 GraphQL 서버의 CPU 점유율이 얼마나 비정상적으로 치솟는지 측정합니다. 반면 gRPC는 얼마나 안정적으로 이를 처리하는지 대조합니다.

<br/>

* **TC 6. 복합 트랜잭션 쓰기 (Data Mutation)**
* **대상:** 신규 주문(`orders`) 생성과 동시에 `order_items`와 `payments` 데이터를 한 번의 요청으로 DB에 동시 삽입(Insert).
* **측정 목적:** 데이터를 읽어올 때뿐만 아니라, 복잡하게 구조화된 페이로드를 서버로 전송하여 쓰기 작업을 할 때 발생하는 파싱 오버헤드를 비교합니다.


---

#### ⚫ Phase 4: 예외 상황 처리 (Edge Cases)

오류가 발생했을 때 각 아키텍처가 에러 메시지를 어떻게 포장해서 돌려주는지 자원 소모를 확인합니다.

* **TC 7. 존재하지 않는 데이터 오류 처리 (Error Handling)**
* **대상:** DB에 존재하지 않는 무작위 가짜 `order_id`로 지속적인 조회 요청을 보내 404 에러를 강제 유발.
* **측정 목적:** REST의 단순한 HTTP 404 반환과 비교하여, 200 OK 상태에서 내부에 Error 배열을 생성해야 하는 GraphQL, 그리고 프록시를 거쳐야 하는 gRPC-Web이 에러 상황에서 서버 자원을 얼마나 소모하는지 측정합니다.
